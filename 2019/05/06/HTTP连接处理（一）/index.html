HTTP连接处理（一）

发表于 2019-05-06 更新于 2019-10-10 分类于 Tcp/Ip

常被误解的Connection首部

HTTP允许在客户端和源端服务器之间存在一串HTTP中间实体（代理、高速缓存等）。可以从客户端开始，逐跳的将HTTP报文经过这些中间设备，转发到源端服务器上去（或者反向代理）

在某些情况下， 两个相邻的HTTP应用程序会为他们共享的连接应用一组选项。HTTP报文中Connection字段有一个由逗号分隔的连接标签列表，这些标签为连接指定了一些不会传播到其他连接中的选项。比如，可以用Connection: close来说明发送完报文之后必须关闭连接。

Connection首部可以承载三种不同类型的标签，因此有时会令人费解:

  • HTTP首部字段名，列出了只与此连接有关的首部
  • 任意标签值，用于描述此连接的非标准选项
  • 值close， 说明操作完成之后关闭这条持久连接

如果连接标签中包含一个HTTP首部字段的名称, 那么这个首部字段就包含了与一些连接相关的信息，不能将其转发出去。在将报文转发出去之前，必须删除Connection首部列出的所有首部字段。

注意 代理会转化Connection首部，这将会造成一些其他问题

串行事务处理时延

如果对连接只是做简单的管理，TCP的性能时延可能会叠加起来。比如，假设有一个包含了3个嵌入图片的WEB页面，浏览器需要发起4给HTTP事务来显示此页面（一个用于HTTP页面，三个用于图片的加载）。如果每个事务都需要（串行地建立）一条连接， 那么连接时延和慢启动时延就会叠加起来

除了串行加载引入的实际时延之外，加载一副图片，页面上其他地方都没有动静也会让你觉得速度很慢。

串行加载的另一个缺点是，有些浏览器在对象加载完毕之前无法获知对象的尺寸，而且他们可能需要对象的尺寸信息来决定对象放在屏幕什么位置上，所以在加载足够多的对象之前，无法在屏幕上显示任何内容。在这种情况下，可能浏览器串行加载对象的进度正常，但用户面对的却是一个空白的屏幕，对加载的进度一无所知。

提交HTTP连接性能的方法

  • 并行连接
    通过多条TCP连接发起并发的HTTP请求

  • 持久连接
    重用TCP连接，以消除连接及关闭延迟

  • 管道化连接
    通过共享TCP连接发起并发HTTP请求

  • 复用的连接
    交替连接请求和响应报文（实验阶段）
